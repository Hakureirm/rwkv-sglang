# Finding F0009: 7.2B scale + dynamic-batch correctness + the comparison

## Hypothesis
The impl should scale to 7.2B (albatross's headline size), dynamic batching should
be correct, and the ours-vs-albatross gap should be characterized at production scale.

## Result

### Correctness — EXACT at every size, dynamic batching safe
- **7.2B** (`rwkv7-g1g-7.2b`, 32L/4096d/64heads, 14.4GB): greedy-match **EXACT 8/8**
  vs the numpy oracle (bf16, cuda-graph ON; lead-verified independently). Fits the
  24GB 3090 (~17.5GB peak, all bsz). 0.1B+1.5B+7.2B all exact.
- **Dynamic-batch correctness**: identical / shared-prefix / mixed batches all exact
  (`bench/verify_batch.py`). Radix-cache bug (B≥3 shared-prefix → corruption; radix-on
  was 0/8) **fixed**: `server_args.py` now registers RWKV7 →
  `_handle_mamba_radix_cache(support_mamba_cache=False)` (mirrors KimiLinear), so radix
  is **auto-disabled** ("Disabling Radix Cache for RWKV7ForCausalLM…") → batch-8 now
  8/8 (lead-verified). Safe-by-default; no launch-flag footgun.

### Speed — ours (sglang, bf16, cuda-graph ON, radix off) vs albatross (3090, fp16) — gap = alb/ours
| size | decode gap (bsz 1/8/32) | prefill gap (bsz 1/8/32) |
|---|---|---|
| 0.1B | 2.6 / 1.7 / 2.5× | 3.4 / 4.9 / 6.2× |
| 1.5B | 2.2 / 1.4 / 1.9× | 1.6 / 1.7 / 1.6× |
| **7.2B** | **1.8 / 1.5 / 1.6×** | **1.4 / 1.2 / 1.2×** |
VRAM @ 7.2B bsz32: **ours 17.5GB (flat in batch) vs albatross ~24GB (near-OOM, grows with B×T)**.

### KEY INSIGHT: the gap shrinks with scale
Tiny models are overhead-dominated (0.1B: ~2.5× decode / up-to-6× prefill); at the
**production-relevant 7.2B the gap is ~1.2–1.8×** AND we use less VRAM AND provide real
dynamic-batch serving (albatross is a static-shape kernel-only micro-bench). The residual
is kernel-quality → M3b (de-FLA kernel vendoring, ADR-0004) closes it further.

## Conclusion
Correctness is fully solid (exact 0.1B/1.5B/7.2B, dynamic batching safe-by-default).
Speed at 7.2B is already within ~1.2–1.8× of albatross with full serving + lower VRAM.
M3b (albatross/own CUDA kernels, removing FLA) is the remaining push to tighten it.
