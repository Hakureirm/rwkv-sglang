# 基准测试全景

本项目所有测过的指标,人类可读版。每张表写明测试条件,并链接已入库的原始输出。
方法学细节和负面结果在 [`findings/`](findings/) 的编号报告里;本页是可以直接读的汇总。

**引擎版本。**自 2026-07-05 起,所有新测量在 **sglang main** 上进行;更早的数字测于
v0.5.10,保留并标注"(v0.5.10)"。两者都有的地方并排展示——版本切换经过完整验证,
正确性没有任何变化,3090 反而各项都略快了。

**两种计时方式**(不同计时方式的数字不能互相比较):

| | 稳态 | 全程 |
|---|---|---|
| 白话含义 | "跑起来以后每秒吐多少字" | "从发出请求到收完全文,平均每秒多少字" |
| 是否包含读题时间(首字延迟)? | 不含 | 含 |
| 工具 | `bench/serving_scale.py` | `bench/bsz_throughput.py`(64 进/256 出) |
| 用于 | §3 单请求阶梯、§4 量化 | §5 并发扫描、§6 全卡、§7 Albatross 对比 |

同一配置同一张卡,稳态读数比全程高约 3%。下面每张表都注明用的是哪种。

所有对比表都用 **cuda-graph 开启**(生产解码路径——serving_scale.py、run_clean_comparison.py、
以及经 serve.sh 的 bsz_throughput.py)。另有一个脚本 throughput.py 报 **eager(cuda-graph 关闭)
基线**,仅供内部内核开发追踪;它的批量解码数字约低 2 倍,从不与 cuda-graph 表格混用。

## 2. 精度指标(RWKV 官方指定的两把尺)

**压缩率**(对新鲜语料的每字节比特数,越低越好;与分词器无关;15 类语料 × 500 篇)。
**2026-07-05 已在 sglang main 上全量重测:下表每个值在 RTX 5090 和 RTX 3090 上都复现到小数点后第四位**
——不同芯片、不同引擎版本,约 750 万 token 的池化交叉熵得到同样的结果(原始件
`bench/results/uncheatable_full_*` 两套):

| 精度 | 压缩率 | 与 fp16 之差 |
|---|---|---|
| fp16 | **0.6085** | — |
| int8 w8g64 | 0.6086 | +0.0001(实际无损) |
| int8 w8a8(高吞吐路径) | 0.6161 | +0.0076 |
| int4 GPTQ | 0.6514 | +0.0429 |
| 7.2B 模型 fp16 | **0.5413** | 比 1.5B 低 0.0672(模型越大压得越好) |
| 7.2B w8a8 | 0.5454 | +0.0041 对 7.2B fp16 |
| 7.2B w4(rwkv_w4,g64) | 0.5615 | +0.0202 对 7.2B fp16 |

**量化在 7.2B 上代价更小。** w8a8:+0.0041(7.2B)对 +0.0076(1.5B);int4:+0.0202(7.2B,普通 RTN
`rwkv_w4`)对 +0.0429(1.5B,更强的 GPTQ)——7.2B 的 RTN 权重即便用更弱的量化器,退化也不到 1.5B GPTQ
的一半,说明大模型显著更能吸收低比特权重(`bench/results/uncheatable_full_{w4,w8a8}_7.2b_5090main.json`)。

15 类逐项明细(全列,不挑好看的)和位置曲线(证明循环状态在持续吸收上下文:开头 3.65
bits,1024 token 之后降到 2.24 bits)见 `bench/results/uncheatable_*` 与对应报告。

**MATH500**(忠实移植 Albatross 的 `eval_math500.py`:同样的提示词、采样参数、判分器、
1500 token 预算):

| 指标 | 值 | 说明 |
|---|---|---|
| avg@64(v0.5.10) | **0.4060**(12,991 / 32,000 次生成) | 500 题 × 每题 64 次 |
| pass@64(v0.5.10) | 0.6980 | 64 次里至少对一次的题占比 |
| 贪心 avg@1,v0.5.10 | 0.3920(196/500) | 确定性解码 |
| 贪心 avg@1,**main** | **0.3940(197/500)** | 差 +0.0020,远小于 ±0.0220 的统计波动 → 精度没有变化(`bench/results/math500_greedy_5090main.json`) |
| avg@64,**main** | **0.4042**(RTX 5090)/ **0.4063**(RTX 3090) | 两张卡都落在旧版 0.4060 的 ±0.0027 单跑波动带内 → 两张卡都无精度回归(原始件 `math500_avg64_{5090main,3090main}.json`) |
| **7.2B** greedy avg@1,main | **0.6320(316/500)** | 旗舰:比 1.5B 的 0.3940 高 23.8pt——推理强得多,5090 上 3,248 tok/s(`math500_greedy_7.2b_5090main.json`) |
| **w8a8** avg@64,main | **0.3812**(12,197/32,000) | 对 fp16 0.4042 = **−2.3pt**——低方差尺量出的真实 int8 推理损失;压缩率(0.6161)和 greedy 都藏住了(`math500_avg64_w8a8_5090main.json`) |
| **w8a8** greedy avg@1,main | 0.3800(190/500) | 对 fp16 0.3940 = −1.4pt(n=500 下在 1 个二项标准误内) |
| **int4 GPTQ** greedy avg@1,main | 0.1560(78/500) | 对 fp16 0.3940 = **−24pt 塌方**——困惑度类尺严重低估 int4 对推理的伤害(见 §4 警告) |

只看压缩率会排成 +0.0001 / +0.0076 / +0.0429——顺序一样但把 int4 的塌方严重低估了,所以
MATH500 才是这里判定量化质量的那把尺。

## 4. 量化(付出什么,得到什么)

三种模式,都配手写 CUDA 核,都能按 GPU 架构即时编译:

| 模式 | 精度代价 | 优势场景 | 权重下载 |
|---|---|---|---|
| int8 w8g64(仅量化权重) | 测不出来(贪心精确;压缩率 +0.0001) | 小批量提速(5090 上单请求比 fp16 全开快 13%)+ 权重体积减半 | ModelScope `Hakureirm/rwkv7-g1-1.5b-w8g64` |
| int8 w8a8(tensor-core) | 压缩率 0.6161(+0.0076,与 cutlass 逐位一致);**MATH500 avg@64 0.3812 对 fp16 0.4042 = −2.3pt**(低方差尺量出压缩率和 greedy 藏住的真实推理损失——与 int4 同类、但轻得多) | 大批量吞吐之王,sm80–90 走上游 cutlass(3090 峰值 9,851 tok/s,64 进/256 出)。Blackwell/sm120 上游没有这个算子;rwkv-sglang 自写的 s8-wmma 核(寄存器分块 V2,位精确门、批不变)已让该档在 5090 端到端可用——int8 GEMM 本身在 M≥512 已快过 fp16 cuBLAS(1.03–1.55 倍),端到端峰值 20,991 tok/s(512 并发,64 进/256 出)= fp16 的 0.9466 倍(剩余差距是逐投影激活量化的 launch 开销,对手是已高度优化的 fp16 基线) | 可联系获取 |
| int4 GPTQ | 7.2B 上 lambada 降 1.28 个百分点(普通 RTN 量化会降 2.64) | 显存最省:7.2B 权重只有 4.6 GB,16 GB 的 T4 上能以 32.9 tok/s 服务 | ModelScope `Hakureirm/rwkv7-g1-{1.5b,7.2b}-w4gptq` |

量化模式需要预量化权重目录(加载器读取 qweight/scale 键;把量化开关指向 fp16 目录会
直接报错,这是有意设计)。

**int8 决定性的地方——单张 32GB 5090 上跑 7.2B(2026-07-06 实测)。** RWKV-7 状态恒定
大小,所以 state pool 的槽位数就是最大并发(每请求状态约 33MB,两者相同——它是 fp32 模型
状态,与权重量化无关)。fp16 权重 14.4GB,pool 只放得下 221 并发,再高就 OOM;w8a8 权重
7.75GB,留出的余量够 640。同一启动配置、cuda-graph 开、64 进/256 出:

| 单卡 7.2B | 最大并发 | 峰值输出吞吐 |
|---|---|---|
| fp16 | 221 | 5,983 tok/s @c192 |
| **w8a8** | **640(2.90×)** | **7,587 tok/s @c640(1.268×,到 640 仍在爬)** |

并发全曲线(输出 tok/s)——fp16 见顶即 OOM,w8a8 继续爬:

| 并发 | 1 | 128 | 192 | 221 | 320 | 448 | 512 | 576 | 640 |
|---|---|---|---|---|---|---|---|---|---|
| fp16 | 124 | 5,668 | 5,983 | 5,747 | ← 221 以上 OOM | | | | |
| **w8a8** | 60 | 4,657 | — | 5,342 | 6,304 | 6,679 | 6,997 | 7,346 | **7,587** |

w8a8 的曲线到 640 仍在上升(撞的是它自己的显存天花板:state pool 20.03GB、剩 1.92GB),
所以 7,587 是显存受限的下限、不是算力见顶。即 int8 让 7.2B 跑到 **2.90× 并发、比 fp16 高
26.8% 峰值——而 fp16 被钉在显存上限**。诚实机制:相同并发≤221 时 fp16 每步更快(不付激活
量化开销);w8a8 完全靠跑到 fp16 物理达不到的并发反超。原始件:`bench/results/72b/`。

**一个诚实的 int4 警告(2026-07-05 实测):**困惑度类指标会低估 int4 对多步推理的伤害。
1.5B 的 GPTQ 权重上,压缩率看起来损失不大(0.6514),但 **MATH500 贪心塌到 0.1560**
(78/500,fp16 是 0.3940)——量化后的模型推理中途跟丢,越写越长直到撞上 token 上限
(截断率 60%,fp16 只有 14%)。结论:1.5B int4 应当作"非推理类任务的省显存工具"使用;
7.2B GPTQ(lambada 损失小得多)是否同样受损正在同尺复测。原始件:
`bench/results/math500_greedy_w4gptq_5090main.json`。

**sm120 w8a8 内核(GEMM 微基准)。**上游 cutlass `int8_scaled_mm` 在 sm120 上编译不了,
所以 Blackwell 消费卡上我们自写的 s8-wmma GEMM(寄存器分块"V2",对逐行参考位精确、批不变)
是唯一的 int8 路。它在 decode/prefill 批量的投影形状上快过 fp16 cuBLAS(RTX 5090,单核 GEMM,
× = 相对 fp16 的加速):

| 投影形状 | M=512 | M=1024 | M=4096 |
|---|---|---|---|
| attn 2048×2048 | 1.08× | 1.33× | 1.52× |
| ffn.k 8192×2048 | 1.45× | 1.52× | 1.55× |
| ffn.v 2048×8192 | 1.03× | 1.28× | 1.53× |

GEMM 赢了,但 1.5B 端到端是 fp16 的 0.9466×(§5):逐 token 的激活量化 launch 无法在每步
约 144 个异构 decode 核间摊销,加上对手是已极致的 fp16 基线,吃掉了内核的余量。这笔税在上面
显存受限的 7.2B 场景里不显(int8 真正的赢——2.90× 并发——就在那里)。原始件:
`bench/verify_w8a8.py --bench`。

## 6b. 多卡:张量并行 / 流水并行(main 上验证,cuda-graph 开)

张量并行与流水并行,在 sglang main 的生产 cuda-graph 路径下测(F0019 的矩阵是 cuda-graph 关)。
1.5B bf16,2×L4,全程计时,**64 进/256 出**(c1/c8/c32/c64=并发)。**tp=2 与 pp=2 均贪心 24/24 == 单卡且确定性**——多卡不改变输出。
(pp 在这里能跑通,先要修一个 cuda-graph capture 崩溃——F0036。)

| 配置 | 贪心 vs 单卡 | c1 | c8 | c32 | c64(峰值) | vs tp=1 |
|---|---|---|---|---|---|---|
| tp=1(1 卡) | 参考 | 72.6 | 482.3 | 1,612.9 | 2,582.6 | — |
| **tp=2** | **24/24 精确** | 105.3 | 655.9 | 2,008.6 | **3,026.2** | **1.17×** |
| **pp=2** | **24/24 精确** | 65.4 | 367.7 | 1,365.5 | 2,288.8 | 0.89× |

诚实读:1.5B 在 PCIe 连接的 L4 上,TP=2 在 c64 给 ~1.17×;PP=2 是 0.89×(此规模流水气泡占主导——
PP 的职责是装下超过单卡的模型,不是给小模型每 token 提速)。价值在于两者在 main 上都**正确且可
生产**;真正需要多卡的模型(7.2B+、NVLink)上的扩展曲线是后续工作。原始件:`bench/results/tppp_l4_main.json`。

## 7. 与 Albatross 对比(BlinkDL 官方速度参照)

Albatross 是一个纯前向测速程序(没有请求调度、动态批和服务接口),所以这个对比只回答
一个问题:单流裸速度差多少。每张卡用同一份 1.5B 权重文件。它的出厂参数是作者在自己的
RTX 5090 上调的——在 5090 上是它的最佳状态,在其他卡上就是用户拿到手的真实状态。
计时说明:Albatross 列不含读题时间,我们这列含(约压低我们 3%),所以下面的比值只会
低估我们,可以放心引用。

| GPU | Albatross(tok/s) | 我们(tok/s) | 我们 / Albatross |
|---|---|---|---|
| T4 | **stock 核编译不过**(sm80+ `cp.async`;可移除——见注†) | 97.1 | 开箱即跑,只我们 |
| L4 | 113.5 | 102.2 | **0.9004** |
| A10G | 203.4 | 168.3 | 0.8274 |
| L40S | 291.8 | 238.0 | 0.8156 |
| A100-40GB | 341.3 | 257.0 | 0.7530 |
| A100-80GB | 385.5 | 278.9 | 0.7235 |
| RTX 3090 | 309.2(我们替它为这张卡重调过参数) | 230.7 | 0.7461 |
| RTX PRO 6000 | 457.4 | 315.0 | 0.6887 |
| RTX 5090(作者自己的调参卡) | 553.9 | 397.3 | 0.7173 |
| H100 | 607.3 | 361.1 | 0.5946 |
| H200 | 684.3 | 399.3 | 0.5835 |
| B200 | 744.0 | 381.6 | 0.5129 |

† T4 这条差距是 Albatross **出厂** WKV 核里的 `cp.async`(sm80+ 指令);BlinkDL 指出这是可移除的
——改核后能在 T4 跑,所以是打包限制而非本质限制。这里的说法严格限定「开箱即用」:我们的栈不改
就能服务 T4,而我们没有基准测过手改后的 Albatross 在 T4 上的数字。

怎么读:差距大小跟显存带宽正相关。推理卡上我们很接近(L4 上 0.9004);带宽极高的卡上
它的整层融合设计越占便宜(B200 上 0.5129),因为我们按算子拆分的实现,每次启动计算核
的固定开销在快卡上占比变大——这正是我们下一步(CUDA graph 消除启动开销、加深融合)的
方向。另外两个事实:我们的 **int4 在作者自己的 5090 上达到它 fp16 速度的 0.9908 倍**
(548.8 对 553.9,跨精度);T4 那行显示了覆盖面的差别。原始件:
`bench/results/albatross_fleet_10cards.json`。

还有一个发现:换到 CUDA 12.9 后,Albatross 出厂参数连在它调参用的 5090 上都不再是

## 7c. 真实负载对比(ShareGPT 变长对话)

上面的合成扫用的是单一固定形状(64 进/256 出)。真实服务是变长的,对调度器的压力不同。
同一个中立客户端(`sglang.bench_serving`)、同一份 ShareGPT 文件、同 500 条请求、同权重,
两引擎各用最优配置。两档负载:峰值(全部请求一次涌入)和稳态(16 req/s)。同等条件的铁证:8 组运行处理的
输入 token 完全相同(168,913 个),生成的输出 token 也完全相同(109,861 个)——同样的
输入进、同样的 token 出(权重相等 + 贪心 + ignore_eos)。

**输出吞吐(tok/s)与延迟,RTX 5090:**

| 负载 | 引擎 | 输出 tok/s | 首字延迟中位 | 每字间隔 p99 |
|---|---|---|---|---|
| 峰值 | rwkv-sglang | **9,602** | **2,503 ms** | **20.5 ms** |
| 峰值 | vllm-rwkv | 8,865 | 3,458 ms | 370.8 ms |
| 16 req/s | rwkv-sglang | 3,300 | 31.6 ms | 37.8 ms |
| 16 req/s | vllm-rwkv | 3,351 | 24.1 ms | 22.7 ms |

**RTX 3090:**

| 负载 | 引擎 | 输出 tok/s | 首字延迟中位 | 每字间隔 p99 |
|---|---|---|---|---|
| 峰值 | rwkv-sglang | **3,974** | **7,297 ms** | **717 ms** |
| 峰值 | vllm-rwkv | 2,805 | 12,750 ms | 1,595 ms |
| 16 req/s* | rwkv-sglang | 2,477 | **316 ms** | 1,239 ms |
| 16 req/s* | vllm-rwkv | 2,600 | 375 ms | 375 ms |

*3090 在这个模型上其实撑不住 16 req/s(两引擎实测都只到约 11–12 req/s),所以这一行是
轻度过载,不是真正的稳态。5090 可以从容处理 16 req/s。

**结论反转——这正是重点。**在*合成固定形状*扫描上,vllm-rwkv 高并发领先(它的 Albatross
内核 + decode-wave 批处理喜欢整齐的形状)。但在*真实变长*负载的峰值下,**rwkv-sglang 两张卡
的吞吐都领先**(5090 上 1.08 倍、3090 上 1.42 倍),且首字延迟中位更低——sglang 的连续动态
批处理能把长短不一的请求打包得更满,不像 wave 调度器在变长形状上留下空泡。稳态 16 req/s 时
两者吞吐相差几个百分点,尾延迟互有胜负(vllm-rwkv 稳态的每字间隔尾更紧;rwkv-sglang 在 5090
峰值下的尾紧得多)。总结:高负载真实混合长度服务下 rwkv-sglang 领先;轻稳态负载下互有来回。
原始件:`bench/results/realload/`。

## 10. 结构性优势:状态大小恒定

| 放大的维度 | 基线 | 放大后 | 显存增量 |
|---|---|---|---|
| 并发 1 → 256 路(1.5B,3090) | 12,420 MiB | 12,622 MiB | **+202 MiB** |
| 上下文 1K → 64K(1.5B) | 12,364 MiB | 12,368 MiB | **+4 MiB** |
| 上下文 1K → 32K(7.2B) | 17,866 MiB | 17,866 MiB | **+0 MiB** |
| 并发 1 → 64 路(7.2B,24 GB 卡) | 46.6 tok/s | 1,802.7 tok/s | +308 MiB |

Transformer 的 KV 缓存在这两个维度上都会增长;RWKV-7 的状态不会。这就是**单张 32 GB 5090
用 w8a8 能同时服务 640 路 7.2B 请求**的原因(§4)——随并发增长的只有 state pool,而它每请求
固定且极小。(上表的显存增长行是 v0.5.10 测量;此性质由架构决定,main 上不变。)

## 11. 投机解码(第一阶段)

小模型一次起草 K 个 token,大模型一次前向验证,被拒绝的部分靠恢复固定大小的状态快照
回滚。现状:功能完整;10 组对照测试中 9 组与普通解码逐 token 相同,平均每轮通过 3.17
个 token(实测起草命中率 0.738)。唯一一处差异已查明原因:它恰好发生在整段文本中两个
候选词概率最接近的位置(差距仅 0.005 nats),属于浮点运算顺序带来的极小抖动——验证走
M=K 的 GEMM,归约顺序与 baseline 的 M=1 解码不同。精确性修复已写好并过门:`gemv_mb`——
一个批不变的 M 行 GEMV,每一行都与解码核(`gemv_m1`)逐位相同,把验证的投影路由到它就能
让"开投机 ≡ 关投机"。剩余:接线、把 worker 移植到 sglang main、加起草/验证的 CUDA graph
(提速那步)。完整分析:[F0031](findings/0031-spec-decode-increment-i.md);可行性 F0029;设计 ADR-0006。

## 12. Apple Silicon(MLX)

Apple Silicon 原生 RWKV-7——MLX + 自写 Metal WKV 核,用与 CUDA **同一个 numpy fp32 oracle** 把关。
MLX 端**已与 CUDA 平台覆盖对齐**:核 profiling、量化、压缩率尺、真实负载基准——全部在 **Apple M5
(32 GB 统一内存,MLX 0.31.2)** 上。这是台共享机(负载 ~8),单流解码有 ±3–5% 抖动;下面的量化头条
数来自**同进程交错 A/B**(每轮 baseline+变体背靠背测,抵消漂移),`bench_mlx.py` 报中位+最优。

### 12.1 fp16 默认——正确性 + 速度(Metal WKV,bf16 权重)

| | 0.1B | 1.5B | 7.2B |
|---|---|---|---|
| 贪心 vs numpy oracle | 24/24 | 24/24 | 8/8¹ |
| 单流解码(tok/s) | 325.6 | 37.3 | 7.5 |
| 读题 1024 token(tok/s) | 11,486 | 1,905 | 441 |
| 峰值内存 | 0.54 GiB | 3.38 GiB | 14.64 GiB |

¹ 7.2B 的 oracle fixture 是 8 token(0.1B/1.5B 是 24)——三个尺寸在**普通算子和 Metal 两条核路径上都逐
token 精确**;7.2B fp16 装进 32 GB 还有余量。融合 Metal WKV 核为默认(读题比普通算子扫描快 4.4–8.1
倍;`RWKV_MLX_WKV=pure` 为免 JIT 回退)。[F0037](findings/0037-mlx-fused-metal-default.md)、[F0038](findings/0038-mlx-m5-kernel-profiling.md)。

### 12.2 M5 硬件天花板 + 单流时间去向(F0038)

实测 M5:**显存带宽 ~123 GB/s**,矩阵 ~11.4 TFLOP/s @2048² / ~13.2 @4096²。bsz1 解码每 token 把全部
权重读一遍(1.5B ≈ **2.88 GB/token**)→ **硬上限 ~42.7 tok/s**,实测 ~33.6 = **达 79%**。所以解码的
杠杆是*更少的权重字节*(量化,§12.3),而非更多 fp16 核调优(把大投影置零的图内消融让解码 34→400
tok/s——解码*就是*那次权重读)。已落核优化:WKV 扫描里 decay 预计算(`exp` 少 D 倍,**位精确**)→
读题 **+0.8% / +1.8% / +3.1%**(0.1B/1.5B/7.2B);读题 chunk 256 近最优。负结果(记录以免重试):T>1
读题编译 +13% 但破坏 0.1B 位精确 → 回退。

### 12.3 量化——MLX 原生 w8g64 / w4g64(F0039),可选;fp16 仍为精确默认

`RWKV_MLX_QUANT=w8|w4`(`mx.quantize` group-64,对齐 CUDA w8g64/w4-g64;仅权重,bf16 激活):

| 模型 | 模式 | 贪心 vs oracle | 解码 tok/s(中位/最优) | 读题 tok/s | 峰值内存 |
|---|---|---|---:|---:|---:|
| 0.1B | fp16 | 24/24 | 325.6 / 331.9 | 11,486 | 0.54 GiB |
| 0.1B | **w8** | **24/24** | **417.3 / 475.7** | 8,458 | 0.43 GiB |
| 0.1B | w4 | 4/24 | 588.1 / 593.4 | 7,831 | 0.36 GiB |
| 1.5B | fp16 | 24/24 | 37.3 / 39.1 | 1,905 | 3.38 GiB |
| 1.5B | **w8** | **24/24** | **55.5 / 56.0** | 1,908 | 2.28 GiB |
| 1.5B | w4 | 24/24² | 94.0 / 95.8 | 1,975 | 1.65 GiB |
| 7.2B | fp16 | 8/8 | 7.5 / 7.9 | 441 | 14.64 GiB |
| 7.2B | **w8** | **8/8** | **12.6 / 12.9** | 484 | 8.88 GiB |
| 7.2B | w4 | 8/8² | 22.0 / 22.9 | 513 | 5.76 GiB |

- **w8 = 贪心无损,推荐量化档**:解码 **+28% / +49% / +68%**(0.1B/1.5B/7.2B),峰值内存 **−20% /
  −33% / −39%**,贪心输出与 fp32 oracle 完全一致。抵消漂移的 1.5B 交错 A/B:fp16 31.4 → **w8 52.0
  (+66%)** → w4 85.1(+171%)。
- **w4 = 省显存 / 极限解码档**:解码 **+81% / +152% / +193%**,峰值 **−33% / −51% / −61%**(7.2B 只
  要 5.76 GiB),代价是真实精度损失(§12.4)。
- ² w4 在 24/8 token 上的贪心吻合是巧合、非无损(0.1B w4 贪心已发散,4/24)。量化下的读题:0.1B −26%
  (小模型读题算力受限→反量化税),1.5B ~持平,7.2B +10%(足够大,读题也部分受带宽限)。
