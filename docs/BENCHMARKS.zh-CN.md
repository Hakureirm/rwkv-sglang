# 基准测试全景

本项目所有测过的指标,人类可读版。每张表写明测试条件,并链接已入库的原始输出。
方法学细节和负面结果在 [`findings/`](findings/) 的编号报告里;本页是可以直接读的汇总。

**模型。**所有权重都是 RWKV-7(RWKV7-G1 系列)。**除非某表/某行另有说明,模型即 1.5B**
(`rwkv7-1.5b-fla`,量化行为其预量化权重)。**0.1B**(`rwkv7-0.1b-fla`)和 **7.2B**
(`rwkv7-7.2b-fla`)行总是标注其尺寸。精度除非行内注明量化档(w8g64 / w8a8 / int4-GPTQ / w4),
否则为 fp16/bf16。

**引擎版本。**自 2026-07-05 起,所有新测量在 **sglang main** 上进行;更早的数字测于
v0.5.10,保留并标注"(v0.5.10)"。两者都有的地方并排展示——版本切换经过完整验证,
正确性没有任何变化,3090 反而各项都略快了。

**两种计时方式**(不同计时方式的数字不能互相比较):

| | 稳态 | 全程 |
|---|---|---|
| 白话含义 | "跑起来以后每秒吐多少字" | "从发出请求到收完全文,平均每秒多少字" |
| 是否包含读题时间(首字延迟)? | 不含 | 含 |
| 工具 | `bench/serving_scale.py` | `bench/bsz_throughput.py`(64 进/256 出) |
| 用于 | §3 单请求阶梯、§4 量化 | §4b int4 速度、§5 并发扫描、§6 全卡、§7 Albatross 对比 |

同一配置同一张卡,稳态读数比全程高约 3%。下面每张表都注明用的是哪种。

所有对比表都用 **cuda-graph 开启**(生产解码路径——serving_scale.py、run_clean_comparison.py、
以及经 serve.sh 的 bsz_throughput.py)。另有一个脚本 throughput.py 报 **eager(cuda-graph 关闭)
基线**,仅供内部内核开发追踪;它的批量解码数字约低 2 倍,从不与 cuda-graph 表格混用。

## 2. 精度指标(RWKV 官方指定的两把尺)

**压缩率**(对新鲜语料的每字节比特数,越低越好;与分词器无关;15 类语料 × 500 篇)。
**2026-07-05 已在 sglang main 上全量重测:下表每个值在 RTX 5090 和 RTX 3090 上都复现到小数点后第四位**
——不同芯片、不同引擎版本,约 750 万 token 的池化交叉熵得到同样的结果(原始件
`bench/results/uncheatable_full_*` 两套):

| 模型 · 精度 | 压缩率 | 与 fp16 之差 |
|---|---|---|
| 1.5B fp16 | **0.6085** | — |
| 1.5B int8 w8g64 | 0.6086 | +0.0001(实际无损) |
| 1.5B int8 w8a8(高吞吐路径) | 0.6161 | +0.0076 |
| 1.5B int4 GPTQ | 0.6514 | +0.0429 |
| 7.2B fp16 | **0.5413** | 比 1.5B fp16 低 0.0672(模型越大压得越好) |
| 7.2B int8 w8a8 | 0.5454 | +0.0041 对 7.2B fp16 |
| 7.2B int4 w4(rwkv_w4,g64) | 0.5615 | +0.0202 对 7.2B fp16 |

**量化在 7.2B 上代价更小。** w8a8:+0.0041(7.2B)对 +0.0076(1.5B);int4:+0.0202(7.2B,普通 RTN
`rwkv_w4`)对 +0.0429(1.5B,更强的 GPTQ)——7.2B 的 RTN 权重即便用更弱的量化器,退化也不到 1.5B GPTQ
的一半,说明大模型显著更能吸收低比特权重(`bench/results/uncheatable_full_{w4,w8a8}_7.2b_5090main.json`)。

**非对称(scale+zero)GPTQ 对 int4 的改良——零核改动、代价便宜(F0043,2026-07-07)。**
在同一份语料(N=300,历史 N=7500 语料已不在盘上,两者不可直接比,仅 fp16 本身在两个 N 下就差
~3%)上做的干净三方重测:

| 1.5B · N=300 | 压缩率 | 与 fp16 之差 |
|---|---|---|
| fp16 | 0.5893 | — |
| int4 GPTQ,对称 | 0.6330 | +0.0437 |
| int4 GPTQ,**非对称** | **0.6186** | +0.0293(补掉对称缺口的 33%) |

核完全不用改:int4 解包本来就用满两补码全范围 `[-8,7]`,对称方案只用了 `[-7,7]`——把
`q_unsigned - 8` 存进去,核的现有输出就已经精确到一个逐组常数的偏差,补一次
`groupsum_x @ bias.T`(O(N·K/64),最宽投影额外算力 ≤1.6%)就能精确修正(`bench/verify_w4.py`
12 组配置验证,误差 7.6e-5~2.2e-4,与核自身噪声地板同量级)。§4 和 F0043 有完整的 MATH500
推理塌方画像——非对称没能完全解决这个,以及为什么暂不推荐上马 Stage 2(真 K-quant 混合精度)。

15 类逐项明细(全列,不挑好看的)和位置曲线(证明循环状态在持续吸收上下文:开头 3.65
bits,1024 token 之后降到 2.24 bits)见 `bench/results/uncheatable_*` 与对应报告。

**MATH500**(忠实移植 Albatross 的 `eval_math500.py`:同样的提示词、采样参数、判分器、
1500 token 预算):

所有行为 **RTX 5090**,除非注明(avg@64 另有 3090 列);"main"/"(v0.5.10)"为引擎版本。

| 模型 · 精度 · 设置 | 值 | 说明 |
|---|---|---|
| 1.5B fp16 avg@64(v0.5.10) | **0.4060**(12,991 / 32,000 次生成) | 500 题 × 每题 64 次 |
| 1.5B fp16 pass@64(v0.5.10) | 0.6980 | 64 次里至少对一次的题占比 |
| 1.5B fp16 贪心 avg@1(v0.5.10) | 0.3920(196/500) | 确定性解码 |
| 1.5B fp16 贪心 avg@1(main) | **0.3940(197/500)** | 差 +0.0020,远小于 ±0.0220 的统计波动 → 精度没有变化(`bench/results/math500_greedy_5090main.json`) |
| 1.5B fp16 avg@64(main) | **0.4042**(RTX 5090)/ **0.4063**(RTX 3090) | 两张卡都落在旧版 0.4060 的 ±0.0027 单跑波动带内 → 两张卡都无精度回归(原始件 `math500_avg64_{5090main,3090main}.json`) |
| **7.2B** fp16 贪心 avg@1(main) | **0.6320(316/500)** | 旗舰:比 1.5B 的 0.3940 高 23.8pt——推理强得多,5090 上 3,248 tok/s(`math500_greedy_7.2b_5090main.json`) |
| 1.5B **w8a8** avg@64(main) | **0.3812**(12,197/32,000) | 对 1.5B fp16 0.4042 = **−2.3pt**——低方差尺量出的真实 int8 推理损失;压缩率(0.6161)和 greedy 都藏住了(`math500_avg64_w8a8_5090main.json`) |
| 1.5B **w8a8** 贪心 avg@1(main) | 0.3800(190/500) | 对 1.5B fp16 0.3940 = −1.4pt(n=500 下在 1 个二项标准误内) |
| 1.5B **int4 GPTQ** 贪心 avg@1(main) | 0.1560(78/500) | 对 1.5B fp16 0.3940 = **−24pt 塌方**——困惑度类尺严重低估 int4 对推理的伤害(见 §4 警告) |
| 1.5B **int4 GPTQ,对称** avg@64(main,3090) | **0.1498**(4,794/32,000) | 对 fp16 avg@64 0.4060 = **−25.6pt**;证实贪心塌方(0.1560)不是解码策略偶然产物——64 次温度采样下量级一致。截断率 57.7%(fp16 14.2%),平均生成 token 1022.9(fp16 581)——"跟丢后停不下来",不是"自信答错" |
| 1.5B **int4 GPTQ,非对称** avg@64(main,3090,新增 2026-07-07) | **0.2199**(7,036/32,000) | 对对称 0.1498 有真实改善(离 fp16 缺口收窄 27.4%,截断率降到 30.5%),来自零核改动的修正(F0043),但离 fp16 仍有巨大差距。这个收窄比例比同一修正在 lambada(35.0%)或压缩率(32.9%)上的收窄都小——指标越关键,单靠加大 scale/zero 编码空间能买回的就越少,是"这不是单纯位数不够"的证据 |

三档量化在**推理尺**上按伤害排序:w8g64(仅权重,贪心无损)→ w8a8(−2.3pt)→ int4(对称或非对称
都是 −24 到 −26pt)。只看压缩率会排成 +0.0001 / +0.0076 / +0.0293–0.0429——顺序一样但把 int4
的塌方严重低估了,所以 MATH500 才是这里判定量化质量的那把尺。

## 4. 量化(付出什么,得到什么)

三种模式,都配手写 CUDA 核,都能按 GPU 架构即时编译:

| 模式 | 精度代价 | 优势场景 | 权重下载 |
|---|---|---|---|
| int8 w8g64(仅量化权重) | 测不出来(贪心精确;压缩率 +0.0001) | 小批量提速(5090 上单请求比 fp16 全开快 13%)+ 权重体积减半 | ModelScope `Hakureirm/rwkv7-g1-1.5b-w8g64` |
| int8 w8a8(tensor-core) | 压缩率 0.6161(+0.0076,与 cutlass 逐位一致);**MATH500 avg@64 0.3812 对 fp16 0.4042 = −2.3pt**(低方差尺量出压缩率和 greedy 藏住的真实推理损失——与 int4 同类、但轻得多) | 大批量吞吐之王,sm80–90 走上游 cutlass(3090 峰值 9,851 tok/s,64 进/256 出)。Blackwell/sm120 上游没有这个算子;rwkv-sglang 自写的 s8-wmma 核(寄存器分块 V2,位精确门、批不变)已让该档在 5090 端到端可用——int8 GEMM 本身在 M≥512 已快过 fp16 cuBLAS(1.03–1.55 倍),端到端峰值 20,991 tok/s(512 并发,64 进/256 出)= fp16 的 0.9466 倍(剩余差距是逐投影激活量化的 launch 开销,对手是已高度优化的 fp16 基线) | 可联系获取 |
| int4 GPTQ | 7.2B 上 lambada 降 1.28 个百分点(普通 RTN 量化会降 2.64);**1.5B 的 MATH500 avg@64 无论对称/非对称都会塌方(对 fp16 −25.6pt / −18.6pt,见下文)** | 显存最省:7.2B 权重只有 4.6 GB,16 GB 的 T4 上能以 32.9 tok/s 服务(2026-07-09 复测 32.8–33.6)+ 单流解码在实测各卡全部最快,fp16 的 1.09–2.61 倍——GPTQ/RTN/fp16 完整速度矩阵见 §4b | ModelScope `Hakureirm/rwkv7-g1-{1.5b,7.2b}-w4gptq` |

量化模式需要预量化权重目录(加载器读取 qweight/scale 键;把量化开关指向 fp16 目录会
直接报错,这是有意设计)。

**int8 决定性的地方——单张 32GB 5090 上跑 7.2B(2026-07-06 实测;fp16 一侧已于
2026-07-07 修正,见 [F0047](docs/findings/0047-fp16-72b-concurrency-correction.md))。**
RWKV-7 状态恒定大小,所以 state pool 的槽位数就是最大并发(每请求状态约 33MB,两者
相同——它是 fp32 模型状态,与权重量化无关)。

**修正说明:** 原 fp16 扫描网格只测到并发 221 就停了,依据是一个从未验证过的假设——
"fp16 在那里已经撞到硬性 OOM 上限"。用更细的网格把 221 以上重新测一遍(同一启动配置,
`--mem-fraction-static 0.85`)发现吞吐其实还在非单调地继续爬升,远没有到顶:fp16 真实
的安全运行上限是**至少 344 并发**(368 能干净启动,但真实请求负载下会 OOM——是 eager
prefill 路径上余量过薄导致的分配器碎片问题,不是 221 处的硬性架构上限),**真实峰值是
6,709 tok/s @ c320**(c336 已确认回落到 6,039,c344 为 6,111)——不是 5,983 @ c192。不过
手写核栈确实比 bf16 stock 路径多占真实显存:同样 `--cuda-graph-max-bs 320` 下,fp16 全核栈
cuda-graph capture 完成后只剩约 2.0GB 余量,而 bf16 stock 路径同配置下有约 6.4GB(权重、
state pool 算法完全相同,只是核栈不同)——所以 fp16 确实无法把并发天花板推得和 bf16 一样
远。但它更快的单步算力足够把这点补回来:fp16 修正后的峰值(6,709 @ c320)仍比 bf16 自己
的峰值(6,171 @ c256,见 §12)高 8.7%,手写核依然是端到端净赢,只是在高并发尾部的优势比
bsz1 时窄一些。同一启动配置、cuda-graph 开、64 进/256 出:

| 单卡 7.2B | 最大并发 | 峰值输出吞吐 |
|---|---|---|
| fp16 | **≥344**(原先误报为 221) | **6,709 tok/s @c320**(原先误报为 5,983 @c192) |
| **w8a8** | **640(1.86×)** | **7,587 tok/s @c640(1.131×,到 640 仍在爬)** |

并发全曲线(输出 tok/s)——fp16 修正后的曲线在 320 见顶后回落(未崩溃);w8a8 靠更大的
显存预算继续爬:

| 并发 | 1 | 128 | 192 | 221 | 256 | 320 | 336 | 344 | 368 |
|---|---|---|---|---|---|---|---|---|---|
| fp16(已修正) | 124 | 5,688–5,695 | 5,999–6,034 | 5,769–5,772 | 6,186–6,205 | **6,709–6,714(峰)** | 6,039 | 6,111 | 真实负载下 OOM |
| **w8a8**(未变) | 60 | 4,657 | — | 5,342 | — | 6,304 | — | — | — |

(fp16 区间取自两次独立复测,`--cuda-graph-max-bs` 分别为 320 与 344,重叠点差异均
<0.5%:`bench/results/72b/sweep_72b_fp16_v2_5090.json` / `sweep_72b_fp16_v3_5090.json`;
原粗网格文件 `sweep_72b_fp16.json` 保留作过程记录,不再是当前结论来源。)w8a8 的曲线到
640 仍在上升(撞的是它自己的显存天花板:state pool 20.03GB、剩 1.92GB),所以 7,587 是
显存受限的下限、不是算力见顶。即 int8 仍然让 7.2B 跑到真实的 **1.86× 并发、比 fp16 高
13.1% 峰值**(相比之前发布的 2.90×/26.8× 下修——是 fp16 一侧被网格测试不足所低估,
w8a8 一侧数字未变)——fp16 依然被钉在显存上限附近,只是没有之前认为的那么紧。诚实机制:
相同并发≤221 时 fp16 每步更快(不付激活量化开销);w8a8 靠跑到 fp16 无法安全维持的并发
反超。原始件:`bench/results/72b/`。

**一个诚实的 int4 警告(2026-07-05 实测,2026-07-07 补上非对称+avg@64,F0043):**
困惑度类指标会低估 int4 对多步推理的伤害。1.5B 的 GPTQ 权重上,压缩率看起来损失不大
(0.6514 旧 N=7500 / 0.6330 干净 N=300 对称版),但 **MATH500 avg@64 是 0.1498**
(32,000 次生成、64 次温度采样——证实贪心读数 0.1560 不是解码策略的偶然产物)**对 fp16
的 0.4060**——量化后的模型推理中途跟丢,越写越长直到撞上 token 上限(截断率 57.7%,
fp16 只有 14.2%,平均生成 1023 vs 581 个 token)。

**非对称(scale+zero)量化——先试的便宜方案,在 1.5B 上确实有用但没解决问题:** avg@64
改善到 **0.2199**(截断率降到 30.5%),补掉 27.4% 的 fp16 缺口。这个收窄比例比同一修正在
lambada(35.0%)或压缩率(32.9%)上的收窄都要小——指标越关键,单靠更宽但仍是 4-bit 的
编码空间能买回的就越少,*在这个尺寸上*读作"这更像长链推理上的误差累积,不是单纯位数
不够"的证据。

**更新(2026-07-08):7.2B 这个尺寸的版本已经落地,而且大幅改写了这个判断。** 同样的协议
(500×64 avg@64)、同一模型家族、参数量翻四倍:

| 7.2B MATH500 avg@64 | 分数 | 相对 fp16 | 截断率 |
|---|---:|---:|---:|
| fp16(基准) | **64.18%** | — | 6.3% |
| **对称 GPTQ int4(推荐档)** | **61.08%** | **−3.1pt** | 14.0% |
| 混合 GPTQ int4(仅 ffn.value+ffn.key 走对称,其余非对称) | 56.03% | −8.15pt | 26.2% |
| 非对称 GPTQ int4(全部矩阵) | 47.78% | −16.4pt | 8.1% |

**−3.1pt,不是 −25.6pt。** 在 1.5B 上,对称 int4 量化让 MATH500 分数掉了超过四分之一;在
7.2B 上,同一套量化方案只掉 3 个百分点。这是对"大模型量化更稳"这个假说的一次戏剧性、
直接的证实——一个 session 之前这还只是一个待验证的假说,现在不是了。**判决:7.2B 尺度上
不需要投入建完整的 K-quant 混合精度重写(Stage 2)**——普通的对称 GPTQ 已经在本项目最硬
的推理尺上接近无损了。

**7.2B 上的非对称尝试——一个真实的发现,不是 bug,也不是能发布的 checkpoint。** 完整非
对称 GPTQ(在 1.5B 上确实*帮了忙*、补掉27%缺口的那个方案)在 7.2B 上反而坍塌了,落在对称
版本之下而不是之上。根因通过矩阵类型消融实验定位,不是靠猜:32.2%的生成退化成一句固定的、
与题目无关的短答案(字面完全相同的"The final answer is 2."原样出现在44个不同题目里)——
这是与内容无关的坍缩,不是渐进式质量下降。系统性排除了8个替代解释,每一条都有独立证据
(加载错了checkpoint、权重损坏、静默RTN回退、通用代码bug——1.5B的非对称路径已证实是好的、
采样/cuda-graph假象、kernel在边界形状上出错、运行时fp16舍入误差,以及"非对称按MSE衡量客观
更差"——它不是:192个被量化矩阵里*每一个*在非对称编码下的重建误差都更低,正符合多一个
zero-point自由度的理论预期)。真正的原因:伤害集中在`ffn.value`和`ffn.key`这两类矩阵上
(只把这两类换回对称、attention保持非对称,在一组精选探测集上能把坍缩率从8/9分别降到1/9
和3/9)——这两类是六种被量化矩阵类型里唯一吃`relu(k)²`这种全非负、86-90%稀疏激活的,和
attention及其余FFN那种近零均值的激活分布结构上不同。这与更早、独立的一次发现
([F0017](findings/0017-w4-int4-quant.md))相呼应——非对称量化以一种纯权重MSE预测不到的
方式系统性伤害这个模型家族,是一个可复现的真实敏感性,不是孤立事件。

**显而易见的修复尝试——构造一个只让这两类矩阵豁免于非对称方案的混合checkpoint——同样没
能达标。** 它相对纯非对称版本确实回收了真实的分数(47.78%→56.03%,+8.25pt,证实了诊断
方向是对的),但依然比纯对称低5个百分点,而且明显地,**截断率反而比两个极端都高**
(26.2%)——这是换了一种失败模式,不是失败变少了:纯非对称checkpoint的退化是快速的、
简短的、与内容无关的坍缩(截断率低,因为一句简短的错误答案能在预算内结束);混合checkpoint
则是推理中途跟丢,拖得很长却给不出结论,更频繁地撞上1,500 token的上限。**结论:7.2B 档
的推荐 int4 checkpoint 依然是纯对称 GPTQ。** 非对称这条调查线值得跑——它产出了一个真实的、
有证据支撑的发现,关于这个架构里哪些矩阵对哪种量化方案敏感——但它没有产出一个值得在这个
尺寸上取代更简单的对称基线来发布的 checkpoint。1.5B 那次塌方看起来更像是小模型特有的脆弱
性,不是量化方案本身有缺陷;int4 的诚实定位是"7.2B+ 上用对称编码是真实的省显存/提速工具,
专门在 1.5B 这个尺寸的推理类任务上用 int8 代替"。1.5B int4(对称或非对称)应当当作这个
尺寸下"非推理类任务的省显存工具"使用,不是通用无损档——w8g64(int8,仅权重)在所有已测
尺寸上都是无损量化档。原始件:
`bench/results/math500_greedy_w4gptq_5090main.json`(1.5B 贪心)、
`bench/results/math500_avg64_7.2b_{fp16,sym,asym,hybrid_ffnvk}.json`(7.2B avg@64)、
`docs/findings/0043-w4-asym-gptq.md`(1.5B 完整画像);7.2B 非对称坍缩+混合方案调查的后续
finding 文档待撰写。

**sm120 w8a8 内核(GEMM 微基准)。**上游 cutlass `int8_scaled_mm` 在 sm120 上编译不了,
所以 Blackwell 消费卡上我们自写的 s8-wmma GEMM(寄存器分块"V2",对逐行参考位精确、批不变)
是唯一的 int8 路。它在 decode/prefill 批量的投影形状上快过 fp16 cuBLAS(RTX 5090,单核 GEMM,
× = 相对 fp16 的加速):

| 投影形状 | M=512 | M=1024 | M=4096 |
|---|---|---|---|
| attn 2048×2048 | 1.08× | 1.33× | 1.52× |
| ffn.k 8192×2048 | 1.45× | 1.52× | 1.55× |
| ffn.v 2048×8192 | 1.03× | 1.28× | 1.53× |

GEMM 赢了,但 1.5B 端到端是 fp16 的 0.9466×(§5):逐 token 的激活量化 launch 无法在每步
约 144 个异构 decode 核间摊销,加上对手是已极致的 fp16 基线,吃掉了内核的余量。这笔税在上面
显存受限的 7.2B 场景里不显(int8 真正的赢——**1.86× 并发**,2026-07-07 按 F0047 更正,见上文
——就在那里)。原始件:
`bench/verify_w8a8.py --bench`。

## 4b. int4 服务速度:GPTQ vs RTN vs fp16(2026-07-09/10 实测)

§4 回答 int4 在精度上*付出什么*;本节回答它*服务起来多快*——起因是 BlinkDL 的直接提问:
对称 GPTQ int4 速度如何?本节所有数据同一协议:64 进/256 出定长扫描,真实服务进程 +
`bench/bsz_throughput.py`(全程计时),cuda-graph 开启并显式设置 `--cuda-graph-max-bs`,
同一时刻每张卡只跑一个服务进程。"门"= 与 fp32 numpy oracle(§1 那把尺)的贪心逐 token
比对,针对的就是产出速度数字的同一套服务配置。

**直接回答:GPTQ 和 RTN 速度一样,在哪都一样。**在所有两者都测过的卡/模型组合上——
5090 的 0.1B/1.5B/7.2B、3090 的 1.5B/7.2B、T4/L4/A10G 的 1.5B、A10G 的 7.2B——每个
成对并发点差异都在 2.2% 以内,多数在 1% 以内(5090 7.2B:九个共同点全部 0.3% 以内;
5090 0.1B:8 点全扫平均 |差| 0.64%;3090 1.5B:六个点全部 1.9% 以内),只有一处例外并
如实圈出:3090 的 7.2B 这对在 c=32/128 处差 3.1–4.2%,RTN 偏快(单流仍在 1.7% 以内
吻合)。同一内核、同一布局——运行时与数值无关,校准值不存在改变速度的机制;3090 7.2B
的漂移我们读作 24 GB 卡饱和状态下的运行间抖动,如实报告而不是取平均抹掉。这正是实现
结构所预言的:两种权重跑的是同一条 `rwkv7_w4` 内核路径、同一种 qweight/scale 布局——
GPTQ 的校准只改张量里的*数值*,运行时一条指令都不变。所以选 GPTQ 还是 RTN 纯粹是精度
问题(§2/§4:GPTQ 赢),从来不是速度取舍。

最优。我们更进一步,**亲手替 Albatross 为这张卡重新调参**(14 处分发表修改,每处都做了
数值与端到端双重验证;抓到并回退了一个微基准假赢面——小权重在缓存里跑赢、真模型冷读时
反而变慢;完整 diff 与证据在 `bench/results/albatross_5090/`)。出厂 → 重调后的提升
(RTX 5090,三跑取中位):

| 模型 | 批量 1 / 8 / 32 | 解码 | 读题 |
|---|---|---|---|
| 0.1b | | +0.0% / **+11.0%** / +0.0% | +0.9% / +2.2% / **+13.4%** |
| 1.5b | | +0.0% / **+6.6%** / **+7.9%** | +5.2% / +1.9% / +2.9% |
| 7.2b | | +0.0% / +3.5% / +0.0% | +1.2% / +2.9% / **+5.0%** |

单流解码纹丝不动(显存带宽墙;出厂状态的 7.2b 147.0 tok/s 本来就超过作者自己发布的
144.04),批量场景最多追回 13%。补一条物理性说明,让 147.0 经得起挑刺:它比稠密 fp16
读带宽下限(13.9 GB/步 ÷ 1.792 TB/s 理论值 ⇒ ≤128.7 tok/s)还高约 14%——这是真的,不是
计时口径问题:Albatross 的 channel-mix 默认走无损稀疏 FFN(`CMIX_SPARSE="no-fc"`,其
README 自称 "sparse FFN (lossless)"),relu² 激活为零的 `ffn.value` 行直接不读。按我们在
7.2B 上实测的 90.2% sqrelu 稀疏度(`docs/design/m6-sparse-ffn.md`),每步实际流量约
10.1 GB ⇒ 有效带宽约 1.48 TB/s——是我们在这张卡上实测 1.69 TB/s 纯读带宽的 87%
(ADR-0008 A0 探针)。我们自己的 bsz1 数字也带同类优化(`scripts/serve.sh` 默认
`RWKV_SPARSE_FFN=1`),所以两家引擎的 bsz1 解码都不该拿稠密读天花板去评判。上面的
单请求比值以出厂值为基准;换成重调后的值,bsz1
处的比值不变(554.0 对 553.9)。我们的启动参数在任何卡、任何 CUDA 版本上都是运行时
现场重选的——这就是下一节量化展示的设计差异。

**带宽差距的进行中工作(2026-07-07,F0051)。** 上表是 F0051/F0052 之前、当时配置(所有
融合开关关闭)的数字。**2026-07-08 起这已不是推荐默认**——见本小节末尾的"提升为默认"说明。
在 H100 上做了一次真实的核 launch 画像(每个完整 decode 步 699 次 launch,不是早前
内部估计的~144——用 `torch.profiler` 重新实测,别信旧数字),找到 launch 数最高的
还未融合核簇(LoRA-output gate 数学:3 个 sigmoid + 周边 elementwise)并融合成一个核,
sm_89 与 sm_90 双架构位精确(`max_abs_diff = 0.0`),开关 `RWKV_FUSED_GATES`(裸环境变量
默认关、纯新增、不影响上表数字——提升为默认的说明见下文)。实测效果(用 bsz8 对照组验证——该配置融合不生效,两张卡都
~0%变化,证明 A/B 隔离干净):**H100 bsz1 359.4→392.6(+9.24%,对 Albatross 比值
0.592×→0.646×);L4 bsz1 82.1→83.1(+1.22%)**——同一份代码改动在 H100 上收益大得多,
本身就是"差距确实与带宽相关"的证据,不只是一个没验证过成因的观察。同一轮画像给出的
诚实上限:即便把剩下所有小 elementwise 核都这样融合,H100 也只能到约 0.69×——更大的
剩余部分是我们的 GEMV 核把 epilogue 当独立 launch 跑,而 Albatross 的巨核把它们直接
内联;那项融合(爆炸半径更高——牵动每个量化档都依赖的 GEMV 核)正在进行。见
[F0051](findings/0051-lora-gate-fusion-highbw.md) 及该序列后续 finding 了解最新状态;
这明确是长期迭代项(task #5),不指望完全补平。

**更新(2026-07-07,F0052)。** 落地并实测了上面提到的第一个"把 elementwise 融进 GEMV
epilogue"候选项:把 FFN 的 `relu(key(xk))**2` 激活(2 次 elementwise launch)直接折进
`ffn.key` 这次 GEMV 自己的写回里,而不是在它之后单独跑 2 个核。sm_89 与 sm_90 上位精确
(`torch.equal`),并验证了量化档(w4、w8a8)在开关两侧位一致、完全不受影响。实测效果
(隔离方法同上):**H100 bsz1 393.2→404.3(+2.82%,独立复测 +2.72%;对 Albatross 比值约
0.65×→0.66×);L4 bsz1 83.1→83.3(+0.24%,噪声量级)**——高带宽卡上是真实收益,低带宽卡
上基本无感,与 F0051 同一条带宽相关性规律再次成立,但收益比 LoRA-gate 融合小得多(与它
折叠的 launch 簇更小成正比,2 次 vs ~7-8 次)。核级别的账目显示融合后的 epilogue 本身几乎
不额外占用 GPU 时间,因此**把上面的上限估计从约 0.69× 上修到约 0.71×**。开关
`RWKV_FUSED_SQRELU`(裸环境变量默认关、纯新增;与 `RWKV_SPARSE_FFN` 按设计互斥——它是
sparse 核自身稠密回退路径上的 epilogue 融合保险,不是叠加在 sparse 之上的第二层收益)。
见 [F0052](findings/0052-sqrelu-epilogue-fusion-highbw.md)。

**提升为默认(2026-07-08)。** 两个开关都各自过了字节级精确门(op 级 + 端到端
`verify_batch.py` greedy-EXACT 对numpy oracle),两张测试卡上零回归,量化档爆炸半径
隔离也确认过。`scripts/serve.sh`(推荐生产启动脚本)现在默认打开两者,并且**同时打开全部
7 个融合开关**重新跑过一遍验证(1.5B fp16,cuda-graph,对 numpy oracle):
`OVERALL: PASS(all batches exact)`——发布的组合是作为整体过的门,不是从各自独立测过的

的问题,不是正确性问题(详见 `scripts/serve.sh` 头部注释的完整论证)。

## 7c. 真实负载对比(ShareGPT 变长对话)

上面的合成扫用的是单一固定形状(64 进/256 出)。真实服务是变长的,对调度器的压力不同。**本节 RWKV-7 1.5B。**
同一个中立客户端(`sglang.bench_serving`)、同一份 ShareGPT 文件、同 500 条请求、同权重,
两引擎各用最优配置。两档负载:峰值(全部请求一次涌入)和稳态(16 req/s)。同等条件的铁证:8 组运行处理的
输入 token 完全相同(168,913 个),生成的输出 token 也完全相同(109,861 个)——同样的
输入进、同样的 token 出(权重相等 + 贪心 + ignore_eos)。

**输出吞吐(tok/s)与延迟,RTX 5090:**

|---|---|---|---|---|
| 峰值 | rwkv-sglang | **9,602** | **2,503 ms** | **20.5 ms** |
| 峰值 | vllm-rwkv | 8,865 | 3,458 ms | 370.8 ms |
| 16 req/s | rwkv-sglang | 3,300 | 31.6 ms | 37.8 ms |
| 16 req/s | vllm-rwkv | 3,351 | 24.1 ms | 22.7 ms |

**RTX 3090:**

| 负载 | 引擎 | 输出 tok/s | 首字延迟中位 | 每字间隔 p99 |
|---|---|---|---|---|
| 峰值 | rwkv-sglang | **3,974** | **7,297 ms** | **717 ms** |

**结论反转——这正是重点。**在*合成固定形状*扫描上,vllm-rwkv 高并发领先(它的 Albatross

批处理能把长短不一的请求打包得更满,不像 wave 调度器在变长形状上留下空泡。稳态 16 req/s 时

峰值下的尾紧得多)。总结:高负载真实混合长度服务下 rwkv-sglang 领先;轻稳态负载下互有来回。
原始件:`bench/results/realload/`。

## 10. 结构性优势:状态大小恒定

| 放大的维度 | 基线 | 放大后 | 显存增量 |
|---|---|---|---|
| 并发 1 → 256 路(1.5B,3090) | 12,420 MiB | 12,622 MiB | **+202 MiB** |
| 上下文 1K → 64K(1.5B) | 12,364 MiB | 12,368 MiB | **+4 MiB** |
| 上下文 1K → 32K(7.2B) | 17,866 MiB | 17,866 MiB | **+0 MiB** |

## 12. Apple Silicon(MLX)

Apple Silicon 原生 RWKV-7——MLX + 自写 Metal WKV 核,用与 CUDA **同一个 numpy fp32 oracle** 把关。
MLX 端**已与 CUDA 平台覆盖对齐**:核 profiling、量化、压缩率尺、真实负载基准——全部在 **Apple M5
(32 GB 统一内存,MLX 0.31.2)** 上。这是台共享机,解码抖动分两种量级:**同一次背靠背会话内**很紧
(~1–5%:2026-07-07 连续跑 10 次,1.5B fp16 解码落在 36.2–36.5 tok/s);但**跨会话**(不同时间/日期,
代码和 `mlx`/`mlx-lm` 版本都相同)时,会话中位数观察到最高 **~12% 峰谷差**——2026-07-06/07 五次独立
1.5B fp16 解码会话横跨 32.8–37.3 tok/s(完整明细见 [F0045](findings/0045-qwen35-mlx-matched-benchmark.md)
的补记)。单次会话的绝对 tok/s 请当作某一时刻的快照、不是恒定常数——下面的量化头条数更可靠,因为它们
用**同进程交错 A/B**(每轮 baseline+变体背靠背测,也能抵消跨会话漂移),`bench_mlx.py` 报中位+最优。

### 12.1 fp16 默认——正确性 + 速度(Metal WKV,bf16 权重)

| | 0.1B | 1.5B | 7.2B |
|---|---|---|---|
| 贪心 vs numpy oracle | 24/24 | 24/24 | 8/8¹ |
| 单流解码(tok/s) | 325.6 | 37.3 | 7.5 |
| 读题 1024 token(tok/s) | 11,486 | 1,905 | 441 |
| 峰值内存 | 0.54 GiB | 3.38 GiB | 14.64 GiB |

¹ 7.2B 的 oracle fixture 是 8 token(0.1B/1.5B 是 24)——三个尺寸在**普通算子和 Metal 两条核路径上都逐
token 精确**;7.2B fp16 装进 32 GB 还有余量。融合 Metal WKV 核为默认(读题比普通算子扫描快 4.4–8.1
倍;`RWKV_MLX_WKV=pure` 为免 JIT 回退)。[F0037](findings/0037-mlx-fused-metal-default.md)、[F0038](findings/0038-mlx-m5-kernel-profiling.md)。

### 12.2 M5 硬件天花板 + 单流时间去向(F0038)

实测 M5:**显存带宽 ~123 GB/s**,矩阵 ~11.4 TFLOP/s @2048² / ~13.2 @4096²。bsz1 解码每 token 把全部
权重读一遍(1.5B ≈ **2.88 GB/token**)→ **硬上限 ~42.7 tok/s**,实测 ~33.6 = **达 79%**。所以解码的
杠杆是*更少的权重字节*(量化,§12.3),而非更多 fp16 核调优(把大投影置零的图内消融让解码 34→400
tok/s——解码*就是*那次权重读)。已落核优化:WKV 扫描里 decay 预计算(`exp` 少 D 倍,**位精确**)→
读题 **+0.8% / +1.8% / +3.1%**(0.1B/1.5B/7.2B);读题 chunk 256 近最优。负结果(记录以免重试):T>1
读题编译 +13% 但破坏 0.1B 位精确 → 回退。

### 12.3 量化——MLX 原生 w8g64 / w4g64(F0039),可选;fp16 仍为精确默认

`RWKV_MLX_QUANT=w8|w4`(`mx.quantize` group-64,对齐 CUDA w8g64/w4-g64;仅权重,bf16 激活):

| 模型 | 模式 | 贪心 vs oracle | 解码 tok/s(中位/最优) | 读题 tok/s | 峰值内存 |

### 12.6 CoreML / Apple 神经网络引擎(ANE)——可行性探测:未通过,不报 tok/s(F0042)

第二条 Apple Silicon 路线的探测——CoreML 走 ANE(不是 GPU)——想作为 §12.1 MLX-GPU 数字之外的第三点
(ANE / GPU / CPU)。**可行性门在建完整模型之前就没通过;按本仓"先过 oracle 门再报速度"的纪律,不报
任何 ANE tok/s 数字。**完整方法学、证据与叫停决定见 [F0042](findings/0042-coreml-ane-feasibility.md),
摘要如下:

被测对象 = RWKV-7 WKV 递推本体(delta-rule 状态更新——真正 RWKV 特有的数值部分,不含周边
Linear/LoRA 投影),直接用 MIL(coremltools 的 IR,不依赖 torch/fla)在真实 checkpoint 几何形状、fp16
下搭建,以 `coremltools.models.compute_plan.MLComputePlan`——CoreML 自己的逐算子设备分配计划——作为
证据(不是猜时间):

| 探测(限定计算单元 CPU_AND_NE,即只许 CPU 或 ANE 二选一) | 算子数 | 偏好=ANE | 偏好=CPU | 不限定(`ALL`)下是否仍是 CPU? |
|---|---:|---:|---:|---|
| WKV 单步,0.1B(H=12,D=64,fp16) | 30 | **0** | 30 | 是,全部 30 个 |
| WKV 单步,1.5B(H=32,D=64,fp16) | 30 | **0** | 30 | 是,全部 30 个 |
| WKV 链式 ×4 步,0.1B(fp16) | 105 | **0** | 105 | 是,全部 105 个 |

两种模型尺寸、两种链长共 168 个非常量算子,**没有一个**曾经被标记 `preferred=ANE`——即便 GPU+ANE
都不限定也一样,CoreML 自己的调度器把整个递推全部路由到 **CPU**。这不是探测方法本身坏掉或者这台机
器根本摸不到 ANE:一个正对照(fp16 批量 GEMM,1024×1024,读题阶段 Linear 投影的那类形状)确实拿到
`preferred=ANE`,**而且**在同一台机器上量到真实的 1.20 倍加速(Apple M5,ANE 可枚举、16 核)。第二
个对照——**解码形状的 GEMV**(`[1,2048]@[2048,2048]`,RWKV 自己的 r/k/v/o 投影在 bsz1 下真实的样
子,也就是任务假设"周边线性层随便哪个框架都能跑好"的那部分)——虽然被标 `preferred=ANE`,实测却比
纯 CPU **慢 18%**(单次、不摊批、不流水线的 ANE 调度,在这个尺寸下摊不平它自己的调度延迟)。

**怎么读**:根子和 §12.2(F0038)一样——bsz1 解码受带宽/发射次数限制,不是算力限制(硬上限约
123 GB/s,MLX-GPU 已跑到 79%);一个为批量矩阵乘设计的加速器,面对"每 token 把权重读一遍、做点小递
推数学"这种工作负载,结构上没什么可加的,而 RWKV-7 的顺序扫描(不做 chunkwise-parallel 重排,为了
保 oracle 位精确)恰恰是对它最不友好的形状。完整的逐层 CoreML 转换器(定长 decode/prefill 程序、显
式状态张量、oracle 门、tok/s)**没有去建**——建了也不会改变这个结构性结论,只会多花时间换来一个套
着 ANE 标签、实际跑在 CPU 上的递推。MLX-GPU(§12.1–§12.5)仍是完整、真正撑得住的 Apple Silicon 故事。

### 12.7 对标——Qwen3.5-2B 在 MLX 上,同协议、同机器(F0045)

Apple Silicon 这一档不是 RWKV 独占的([F0044](findings/0044-qwen35-mlx-feasibility.md)):Qwen3.5-2B
靠 `mlx-lm` 0.31.3 自带的原生实现就能跑在 MLX 上(其 Gated-DeltaNet 层有一个自写的真 Metal 核,不是

两边的"bf16"行确实都是 bfloat16——RWKV-7 自己在 §12.1 表头写的"fp16"是沿用本项目 CUDA 基线的命名
习惯,不是字面上的 float16(完整说明见 F0045)。int4 两边用的是同一套 MLX 原生 group-64 affine
仅权重量化方案(RWKV-7:`RWKV_MLX_QUANT=w4`;Qwen3.5:`mlx_lm.convert -q --q-bits 4
--q-group-size 64`)。

| 模型 | 精度 | 解码 tok/s(中位/最优) | 读题 tok/s(1024 token) | 峰值内存 |
|---|---|---:|---:|---:|
| RWKV-7 1.5B | bf16 | 32.8 / 33.6 | 1,691.6 | 3.38 GiB |
| Qwen3.5-2B | bf16 | 27.5 / 27.7 | **2,800.5** | 4.65 GiB |
| RWKV-7 1.5B | int4(w4g64) | 89.5 / 91.8 | 1,913.2 | **1.65 GiB** |
| Qwen3.5-2B | int4(g64) | 89.3 / 89.9 | **2,691.3** | 2.28 GiB |

同一次会话背靠背测得——两个模型同一轮跑、同样的负载条件。F0045 里还有一份可复现性检查(两边各自
独立重测一次,差幅 0.4–4.9%,落在本文档记录的 ±3–5% 抖动带内)以及规范引用(§12.3 的 RWKV-7 bf16
解码 37.3 tok/s,是更早一次会话测得的)作为交叉参照。

**split 结果,不藏着掖着**:RWKV-7 在 bf16 上赢单流解码(中位 **+19.3%**,32.8 对 27.5),int4 上是
统计意义上的平手(**+0.2%**,89.5 对 89.3——落在跑次间噪声范围内;换成引用 §12.3 的规范数字则差距

整个 prompt 窗口上吃到密集批量矩阵乘的并行度,而 RWKV-7 顺序的 WKV 递推即便分块也追不上。RWKV-7 在
两档都少约 27% 峰值内存,这在很大程度上是因为它标称就是个更小的模型(1.5B 对 2B),而非架构效率上
的胜利。完整方法学、可复现性检查、诚实的局限(Qwen3.5 没有正确性 oracle、这里也没跑压缩率尺)见
[F0045](findings/0045-qwen35-mlx-matched-benchmark.md)。

## 13. 与 Qwen3.5 对比(同引擎、同精度、尺寸匹配)

Qwen3.5(阿里巴巴)是**混合 Gated-DeltaNet** 架构——既有我们自己的实测确认(在自家 sglang
装置上 bring-up:24 层,18 层线性attention/GDN + 6 层全 attention,`interval=4`),也有
BlinkDL 自己参考实现的独立印证(`RWKV-LM/RWKV-v7/run_rwkv7_qwen35.py` 编码了完全相同的
`i%4!=3` 分层规则)。GDN 和 RWKV-7 自己的递推都是 DPLR(对角加低秩)族的成员——这是同代际
同族对比,不是"新老架构对撞"。Bo Peng 自己给出的 state 参数量公式把结构性差异说得很明白:
RWKV-7 的 state 是 `L·(2D + 64D)`,与上下文长度无关的常数;Qwen3.5 是
`L·¾·(3·6D + 2·128D) + L·¼·(2·2·256·T)`——第二项正比于 T,因为它 24 层里有 6 层是真
attention,需要随长度增长的真 KV cache。下面每一个并发天花板结果都能追溯到这一条结构事实。

**尺寸匹配**(最接近的可用 Qwen3.5 档位;尺寸并不相等,两边都全程标注):

| RWKV-7 | Qwen3.5 | 说明 |
|---|---|---|
| 1.5B | 2B | 主战场 |
| 7.2B | 9B | 主战场,int8/状态大小优势最明显的档位 |

**为不相等的尺寸做归一化。** 两个档位的尺寸差全程如实标注,但"总参数量"高估了 decode
实际工作量的差距。稳态 decode 每生成一个 token 就把模型主体完整过一遍(小批量下带宽受限:
读的字节数 ∝ 参数量;大批量下算力受限:FLOPs ∝ 参数量),而 embedding 表每步只贡献一次
单行查表,Qwen3.5 checkpoint 里自带的视觉塔和 MTP 块在纯文本基准里根本不会被执行。所以
跨尺寸读速度,正确的分母是非 embedding 参数量——跨模型族比尺寸的标准口径,两边对称地把
两个词表矩阵都排除在外(logits 矩阵每步确实会被完整触碰一次,下面用另一种记账口径量化
它,不是装看不见)。精确账目,方法:直接解析每个被服务 checkpoint 的 safetensors 文件头
(主机侧读文件头,不加载权重),对张量形状逐个求积求和,按 key 分组为 embedding /
lm_head / 视觉 / MTP / 主体:

| 模型 | 总参数 | embedding | lm_head | 视觉+MTP(本对比中闲置) | 非 emb(decode 实际触碰) | 档内比例 |
|---|---:|---:|---:|---:|---:|---:|
| RWKV-7 1.5B | 1.527B | 0.134B | 0.134B | — | **1.259B**(1,258,969,088) | 1× |
| Qwen3.5-2B | 2.274B | 0.509B | 共享 embedding,只计一次 | 0.392B | **1.373B**(1,373,265,728) | 1.091× |
| RWKV-7 7.2B | 7.199B | 0.268B | 0.268B | — | **6.662B**(6,662,270,976) | 1× |
| Qwen3.5-9B | 9.653B | 1.017B | 1.017B | 0.699B | **6.920B**(6,919,565,824) | 1.039× |

decode 实际触碰的权重差距(1.091×、1.039×)远小于总参数差距(1.489×、1.341×):名义尺寸
差里的大头是词表(Qwen3.5 V=248,320 对 RWKV-7 V=65,536)加上本对比中闲置的视觉/MTP 权重。
把 §13.1 的同精度 bf16 头条数字按尺寸校正吞吐重读(tok/s × 非 emb 参数量,单位 B·tok/s
——每生成一个 token 就把每个 decode 权重应用一次,所以这个量在小批量下正比于持续字节
带宽、大批量下正比于 FLOP/s):

| 档位 | 指标 | RWKV-7 原始 → 归一化 | Qwen3.5 原始 → 归一化 | 归一化判定(原始判定) |
|---|---|---:|---:|---|
| 1.5B/2B | 单流 | 256.6 → 323.1 | 335.8–336.3 → 461.1–461.8 | Qwen3.5 **+42.9%**(原始 +31.0%) |
| 1.5B/2B | 峰值 | 21,431.9 → 26,982 | 17,587.0 → 24,152 | RWKV-7 **+11.7%**(原始 +21.9%) |
| 7.2B/9B | 单流 | 87.7 → 584.3 | 96.0 → 664.3 | Qwen3.5 **+13.7%**(原始 +9.4%) |
| 7.2B/9B | 峰值 | 6,171.3 → 41,115 | 4,295.8 → 29,725 | RWKV-7 **+38.3%**(原始 +43.7%) |

归一化让每一条差距都朝 Qwen3.5 方向移动,但没有翻转任何判定:Qwen3.5 的单流领先扩大
(它每步搬动的 decode 权重更多、还照样更快),RWKV-7 的峰值并发领先收窄但保住。同样的
×1.091/×1.039 换算适用于本节任何同精度行——例如 §13.2 的 3090 1.5B/2B 峰值差距(原始
+11.7%)归一化后收窄到 +2.4%,接近打平。有一个记账口径的选择如实摆出来、不埋起来:
logits 投影矩阵每生成一个 token 确实要被完整读/算一遍,如果把它也计入分母(RWKV-7 加
0.134B/0.268B,Qwen3.5 加 0.509B/1.017B——3.8 倍词表就是 3.8 倍 logits 矩阵,共享
embedding 省的是显存、省不掉每步的读取),1.5B/2B 档的归一化峰值判定会翻转为 Qwen3.5
+10.8%,而 7.2B/9B 档峰值仍是 RWKV-7 +25.5%。也就是说,小档位的"每参数"峰值判定对
记账口径敏感;7.2B/9B 档在两种口径下都稳。本节的主张仍以未归一化的原始数字为准——这
一块是尺寸公平性的交叉核验,不是替代。

权重:同引擎(sglang main,原生 `qwen3_5.py` 支持,不用改分支)、每行同精度。这同时是
一次"双方都在最强可用配置"的对比,不是弱基线对比:双方共用的 sglang 正是 Qwen3.5 自己
模型卡为生产/高吞吐场景"强烈推荐"的部署引擎之一——Qwen3.5 跑的是它厂商背书的一等路径,
不是被拽到陌生引擎上比。**Qwen3.5
必须用 `--dtype bfloat16`**——`float16` 会在一个混合 SSM 通用 Triton 核
(`causal_conv1d_triton.py`,Mamba2/Qwen3-Next/Qwen3.5 通用,不是我们该碰的东西)的
dtype 分支上崩溃,改用 checkpoint 原生的 bf16 即可绕开。这意味着本文档别处 RWKV-7 的
**fp16** 头条数字**不能**直接跟 Qwen3.5 的 bf16 数字比——下面每张表要么是 bf16 对 bf16,
要么明确说明是在引用 RWKV 的 fp16 优化数字以及原因(RWKV 的手写核只吃 fp16;bf16 走的是
RWKV 的原生/未优化路径)。

### 13.2 桌面档——RTX 3090

### 13.3 Apple Silicon 档——MLX

层有真正的、非占位的 Metal 核,不是退化路径(F0044)。同协议、同机器的多轮匹配对比,

模式的截断率(2B档52.4%)照实标注,不折进一个数字里藏起来:支撑这个数字的 pilot 试跑
扫描(F0053)在测过的每一档 token 预算(最高到 16,384)上都还在往上涨,而 32,768

上限。9B 档的思考模式那一行本来也打算沿用同样的"披露下限"处理——pilot 在同一 16,384
预算下截断率约13%,比2B档pilot早期读数窄一些——但**完整的 500×64 测算跑到 25%
(8,000/32,000 rollouts)时被主动叫停,没有跑完。** 实时监控发现:思考模式的长推理链
会让每条请求占用的 KV-cache 随着测算推进持续变大,把实际并发从~128压到~22,聚合吞吐随
时间结构性下滑(不是卡死,是真实的显存约束)——按这个规律重新推算,跑完要好几天,而这
张卡是本项目刚采纳的 SkyPilot 共享调度机制下、同一台机器上另一个项目正在等着用的资源。
既然上面的非思考模式数字已经是这个档位的头条数字、思考模式本来就是次要项,为了追这个
数字占着共享的卡好几天不划算——评测被主动杀掉、卡已经还回去,没有硬撑跑完。这一行如实
留空,不拿旧数字或外推数字凑数;以后用本项目新采纳的 `sky launch` 机制重跑补齐,是留着
的合理后续动作,不是关掉的门。

**完整图景需要把这张表和 §7 的峰值并发吞吐表放在一起看,不能只读其中一半。** 在这里测的
准确率轴上,Qwen3.5 在两个已测尺寸都领先。但在峰值并发吞吐——本项目自己的北极星指标,

bf16 峰值:1.5B/2B档 +21.9%,7.2B/9B档 +43.7%)。这是两个不同的能力维度,不是需要往
哪边"扳平"的矛盾;像"RWKV更快"或者"Qwen3.5更强"这种笼统说法,不说清楚是哪个轴,
都是不完整的。

### 13.5 正确性

这次对比里 Qwen3.5 的输出不只是"看着连贯"——两个已测尺寸都用 BlinkDL 自己独立的
numpy fp32 参考实现做过 oracle 门(与本项目一直对自己 RWKV-7 核用的同一套严格标准)。
Qwen3.5-**2B**:在这次对比用到的两条服务路径上都验证过——mlx-lm(bf16)和 sglang(bf16)
在探测 prompt 的 top-1/top-5 下一 token 分布上都与参考实现一致,差异符合正常 bf16
舍入(F0050)。Qwen3.5-**9B**:本项目实际发布 9B 数字所用的 sglang(bf16)服务路径,
也和同一份参考实现一致——top-1精确匹配、top-10 token集合10/10完全相同、最大绝对概率差
0.0049(F0054)。让 9B 的门过关不是简单套新常数就行——9B 的线性注意力层有一个非对称的
key/query 对 value 头数(16对32),upstream自己的参考脚本那句`np.split(qkv, 3)`(要求三等
分)根本跑不起来;根因追到 HF transformers 自己的 `Qwen3_5GatedDeltaNet.forward()`,用
GQA式的头扩展修好——这是2B/0.8B档从未需要面对的情况(它们的key/value头数正好相等,这也
正是F0050当初没能发现这个问题的原因)。这个修复经过两种方式验证才被采信:代数上(手动
推导出与HF自己参考递归公式等价)和实证上(修改后的代码重新跑对2B checkpoint,精确复现
F0050已发布结果到5-6位有效数字——是严格的泛化,不是恰好也对2B有效的重写)。
