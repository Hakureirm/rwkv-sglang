## 现在在做(2026-07-08,跑到一半时更新)

桌面档 Qwen3.5-9B 并发搜索**已完成**(F0049,RWKV-7 确认平坦的峰值比 Qwen3.5-9B 被显存墙截断的
峰值高 27.0%-30.5%)。2026-07-07 推送前审计发现的 PII/内网标识泄露**已彻底修复**
(`git filter-repo` + force-push,全历史核实零残留)——细节见
`memory/feedback-scrub-infra-identifiers-precommit.md`(如果你有权限访问的话)。高带宽卡的
epilogue 融合核(F0052)**已完成并发布**:L4 和 H100 上位精确门禁均通过,拿到真实(虽然不大)的
提升——H100 bsz1 +2.82%(393.2→404.3 tok/s),L4 +0.24%(噪声量级,小规模融合本就该这样),
F0051 的 H100 GPU-忙碌时间理论上限估计从 0.69× 上调到 0.71×,已提交(`3f9053e`)并推送,
`RWKV_FUSED_SQRELU` 保持默认关闭。

眼下的关键路径是两个长跑任务,都很健康,都是因为任务体量大所以要跑好几个小时
(500 道 MATH500 题 × 64 次采样 = 各自 32,000 次生成):

- **3090 箱**:7.2B int4-GPTQ MATH500 avg@64 全套流程。校准(192/192 个 Hessian 分片,拆成
  4 份跑以塞进显存)和非对称量化**已完成**——中途撞上一个真实的阻塞点并已修复(管线的临时
  工作目录里躺着一份 F0043 之前的旧版 `gptq_w4.py`,没有 `--asym` 支持;已从主仓库刷新过来,
  细节见 `memory/project-rwkv-w4-quant.md`)。Oracle 抽检已通过。**目前在跑 fp16 baseline 的
  avg@64**(sym/asym 要对照的基准点)——这正是 F0043 要求的直接后续,决定要不要投入建一套
  完整的 K-quant 混合精度(Stage 2)。还没跑完;fp16 之后还要跑 sym 和 asym。
- **塔机(RTX 5090)**:Qwen3.5-2B MATH500 avg@64。第一次全量跑(chatml_thinking 模式)出来的
  结果**93.15% 的生成被截断**——这是一个真实的方法学 bug,不是一个结果:harness 沿用了 RWKV
  自己的采样默认值(`top_p=0.28`),而不是 Qwen3.5 官方文档写明的 thinking 模式推荐参数,而且
  压根没有 `presence_penalty` 支持(Qwen3.5 官方文档 thinking 模式推荐 `presence_penalty=1.5`)。
  已修复(加上 presence_penalty 支持、按模式纠正采样参数),并在几个不同 token 预算下做了小规模
  pilot 摸底,再决定全量重跑用多大预算。**non-thinking 模式已经跑完且数字干净**:准确率
  67.6%,截断率 0.99%(这是 Qwen3.5-2B 官方文档写明的默认模式——一个真实的、对 RWKV 不利的
  结果,相对 RWKV-7 1.5B 已发布的 40.4-40.6%,如实记录,不回避)。**thinking 模式正在跑**,
  用的是一个封顶预算(mnt=16384,是从模型官方文档默认值 mnt=32768 主动降下来的——pilot 数据
  显示截断率长尾分布,跑满 32768 的代价大到不现实,详见 `memory/project-qwen35-benchmark.md`
  第十一轮的 pilot 扫描数据)。

本机 Mac 按设计闲置 GPU 重活(Apple Silicon 上跑 Qwen3.5-MATH500 那次撞上真实内存压力,
按明确指示停下,没有重试)——目前只用于编排/监控,包括一个小型本地 Rust 仪表盘
(`../rwkv-lab-dashboard/`,放在本仓库之外)实时看两台机器的 GPU/进程/日志状态,不用每次都手动
SSH 上去查。

## 排队等待(等上面的落地后)

1. **7.2B int4 MATH500 结果 → w4 Stage 2(K-quant 混合精度)的真正 go/no-go 决策**。这是
   "接近 Q4_K_M"的真正工作——超块=256、子块=32、6-bit scale+min、按张量角色分 4/6-bit 混合精度。
   **还没开始**——这是一个真实的决策点,不是走过场:F0043 在 1.5B 上的数据显示,非对称量化单独
   只能收窄 27-35% 的 fp16 差距(越关键的指标收窄得越少),而且 MATH500 avg@64 的塌方幅度远超
   lambada 暗示的量级,这暗示问题可能不是纯粹的位预算问题,换个更花哨的编码方式未必能解决。
   3090 上正在跑的 fp16/sym/asym 7.2B avg@64 数字正是这个决策需要的数据——"先测量,别盲目开工"。
2. **Qwen3.5 对比 → 真正写进 BENCHMARKS 大章节。** 所有拼图(云档速度、桌面档速度、Apple
   Silicon 档速度、云档精度、oracle 门禁,再加上 thinking 模式落地后修正过的 MATH500 avg@64
   数字)都要整合进 `docs/BENCHMARKS.md`/`.zh-CN.md` §13.4 一个连贯、互相引用的章节——目前仍标
   "进行中",等 thinking 模式的结果。
3. **高带宽卡内核工作的下一步。** F0052 拿到了真实但不大的提升;按 F0051/F0052 自己的上限分析,
   下一个候选是 128-bit 向量化 GEMV load——还没尝试,目前没有 agent 在跟进。
4. **投机解码速度剖析**——F0046 留了三个未验证的假设(逐层 state `.clone()` 的开销、
   target-verify 的开销、逐轮 Python 编排的开销),还没人测过到底是哪个占大头。优先级低于上面
   三项(本项目一贯的"全场景优先于单流"原则把并发/大批量工作排在单流投机解码速度前面),
   但没有被放弃。

## 关于硬件访问方式的说明

本项目用三类算力:两台长期持有/租用的机器(一台工作站级 GPU、一台桌面级 GPU,本仓库其他地方
按角色称呼,不点具体厂商名字)、本机 Mac 做 Apple Silicon 相关工作、以及针对特定架构短租的卡
(比如高带宽的数据中心级卡),仅在某个问题确实需要本项目不拥有的硬件时才短租——所有已提交材料
一律表述为"高带宽 GPU"或"各卡自己的真实硬件",绝不点出租用方名字,这是本项目一贯的发布纪律。
如果你在读这份文件想搞清楚为什么某些 findings 引用了 H100、B200 这类架构,却没有配上本项目
通常那两台机器的名字——原因就在这里。
